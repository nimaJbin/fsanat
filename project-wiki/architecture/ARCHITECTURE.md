# معرفی پروژه

Fater Sanat Javan یک پلتفرم e-commerce اختصاصی برای دامنه Fsanat.ir است. این پروژه باید به عنوان یک سیستم تجاری واقعی، مقیاس‌پذیر و قابل توسعه طراحی شود، نه فقط یک فروشگاه آنلاین ساده.

هدف معماری پروژه ایجاد یک ساختار modular، قابل نگهداری، Dockerized و آماده رشد است تا بتواند در آینده ماژول‌هایی مانند مدیریت محصول، انبار، سفارش، پرداخت، محاسبه سود، گزارش‌گیری، SEO، اتصال حسابداری و سرویس‌های عملیاتی را پشتیبانی کند.

اولین هدف عملی پروژه، راه‌اندازی زیرساخت پایه شامل Authentication، Admin Panel، User Management و ساختار فنی تمیز برای توسعه ماژول‌های بعدی است.

# نقش‌های پروژه

## Nima

Nima مسئول بخش فنی پروژه است. این مسئولیت شامل source code، software architecture، طراحی سیستم‌ها، پیاده‌سازی، توسعه backend و frontend، ساخت Admin/User panels، اتصال سرویس‌ها، نگهداری فنی و تصمیم‌گیری‌های مرتبط با معماری نرم‌افزار است.

Nima همچنین مسئول مستندسازی فنی، هماهنگ نگه داشتن مستندات با تغییرات پروژه و طراحی ساختارهایی است که توسعه بلندمدت را ممکن کنند.

## Peyman

Peyman مسئول بخش مالکیتی، تجاری و عملیاتی پروژه است. این مسئولیت شامل brand، domain، server، hosting، payment gateway، موجودی فیزیکی، عملیات انبار، بسته‌بندی، ارسال، تامین کالا و هزینه‌های عملیاتی کسب‌وکار است.

Peyman قوانین، نیازها و محدودیت‌های تجاری پروژه را مشخص می‌کند و این موارد باید در طراحی سیستم‌ها و مستندات مرتبط منعکس شوند.

# مالکیت‌ها

مرزبندی مالکیت‌ها باید همیشه در مستندات، تصمیم‌های فنی و طراحی سیستم‌ها شفاف باقی بماند.

- Brand، domain، server، hosting، payment gateway، موجودی فیزیکی، عملیات لجستیک، بسته‌بندی، ارسال و دارایی‌های فیزیکی کسب‌وکار متعلق به Peyman هستند.
- Source code، software architecture، technical implementation، development process، منطق پلتفرم دیجیتال و سیستم‌های فنی متعلق به Nima هستند.

این تفکیک برای جلوگیری از اختلاط مسئولیت‌های مالی، تجاری، عملیاتی و فنی ضروری است.

# مسیر فنی پروژه

Laravel/PHP هسته اصلی backend پروژه خواهد بود، چون برای Authentication، admin workflows، queues، jobs، events، policies، services، database migrations و پیاده‌سازی منطق تجاری قابل نگهداری، بستر بالغ و مناسبی فراهم می‌کند.

زیرساخت پروژه باید Dockerized باشد تا محیط توسعه، تست، استقرار و توسعه آینده قابل پیش‌بینی و قابل تکرار باشد.

Redis باید برای cache، queues، locks، rate limiting و نیازهای fast shared-state استفاده شود.

انتخاب database رابطه‌ای بین MySQL و PostgreSQL در مرحله مناسب و بر اساس محدودیت‌های hosting، نیازهای عملیاتی، یکپارچگی‌ها و ترجیح تیم نهایی می‌شود.

Node.js در آینده فقط برای بخش‌هایی استفاده می‌شود که مزیت فنی روشن داشته باشند؛ مانند realtime features، automation، workers، frontend build tooling یا سرویس‌های مستقل آینده. Node.js نباید بدون دلیل فنی مشخص به عنوان backend اصلی دوم وارد معماری شود.

Admin Panel و User Panel از بخش‌های اصلی پلتفرم هستند و باید از ابتدا در طراحی معماری لحاظ شوند.

معماری باید قابلیت رشد در تعداد کاربران، محصولات، سفارش‌ها، گزارش‌ها، سرویس‌ها و integrationهای آینده را داشته باشد.

# اصول معماری

- معماری باید Modular باشد تا هر حوزه تجاری بتواند مستقل‌تر توسعه پیدا کند.
- کد باید بر اساس اصول Clean Code نوشته شود.
- سیستم‌ها باید reusable طراحی شوند و از تکرار منطق جلوگیری شود.
- طراحی باید expandable باشد و امکان افزودن، حذف یا تغییر ماژول‌ها را در آینده فراهم کند.
- روند توسعه باید documentation-first باشد؛ یعنی سیستم‌های مهم قبل از پیاده‌سازی یا همزمان با آن مستند شوند و بعد از تغییر بروزرسانی شوند.
- تمام منطق‌های مالی باید traceable و auditable باشند.
- Business Logic باید از Controller، UI و لایه‌های ارائه جدا شود.
- قوانین تجاری باید در ساختارهای مشخص، قابل تست و قابل نگهداری قرار بگیرند.
- تصمیم‌های مهم معماری باید در مستندات ثبت شوند.

# سیستم محاسبه سود

سیستم Profit Calculation فقط باید فروش‌هایی را در نظر بگیرد که از طریق payment gateway وب‌سایت تکمیل شده‌اند. فروش‌های فروشگاه فیزیکی، فروش‌های خارج از سایت و تراکنش‌هایی که وارد چرخه رسمی فروش وب‌سایت نشده‌اند، نباید در سود مشترک وب‌سایت محاسبه شوند.

هزینه پایه محصول باید هنگام ورود محصول به چرخه فروش وب‌سایت ثبت و حفظ شود. افزایش قیمت بازار یا تورم آینده نباید به صورت مصنوعی باعث افزایش سود مشترک شود.

محاسبه سود باید بر اساس فروش واقعی وب‌سایت، هزینه پایه محصول و هزینه‌های تاییدشده مرتبط با وب‌سایت انجام شود.

ثبت دستی هزینه‌ها باید پشتیبانی شود، اما فقط هزینه‌های تاییدشده می‌توانند وارد محاسبه سود شوند. هزینه‌های بالاتر از 15,000,000 تومان، به جز هزینه خرید محصول، نیازمند تایید هر دو طرف هستند.

# سیستم Targeting Engine

Targeting Engine یک سیستم reusable برای انتخاب محدوده اثرگذاری قوانین و عملیات مختلف است. این سیستم باید بتواند هدف‌گذاری را در سطوح زیر پشتیبانی کند:

- whole site
- store only
- category
- subcategory
- specific product

این engine در آینده باید توسط سیستم‌های مختلف استفاده شود، از جمله:

- payment disabling
- bulk pricing
- discounts
- campaigns
- operational controls

هدف از طراحی Targeting Engine جلوگیری از تکرار منطق هدف‌گذاری در ماژول‌های مختلف و ایجاد یک ساختار مشترک، قابل توسعه و قابل استفاده مجدد است.
