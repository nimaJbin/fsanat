# زبان مستندات

- تمام فایل‌های مستندات پروژه باید به زبان فارسی نوشته شوند.
- استفاده از اصطلاحات فنی انگلیسی در کنار متن فارسی مجاز است.
- نام کلاس‌ها، سرویس‌ها، متدها، پکیج‌ها، تکنولوژی‌ها، فریم‌ورک‌ها، APIها، پایگاه داده‌ها و ساختارهای فنی می‌توانند انگلیسی باقی بمانند.
- توضیحات باید واضح، خوانا، حرفه‌ای، ساختارمند و مناسب توسعه بلندمدت پروژه باشند.

# Owner Draft Rules

این بخش برای ثبت قوانین، توضیحات و یادداشت‌های مستقیم مالک پروژه در نظر گرفته شده است.

- مالک پروژه ممکن است قوانین یا توضیحات جدید را به فارسی وارد کند.
- Codex موظف است قبل از شروع هر تسک این بخش را مطالعه کند.
- قوانین جدید باید در معماری، طراحی سیستم‌ها، تصمیم‌های فنی و مستندات مرتبط اعمال شوند.
- Codex باید قوانین جدید را در فایل‌های مرتبط ثبت، هماهنگ و به زبان مستندات پروژه بازنویسی کند.
- قوانین فارسی مالک نباید حذف شوند مگر با درخواست مستقیم مالک پروژه.
- اگر قانون جدیدی با مستندات موجود تعارض داشت، Codex باید مستندات مرتبط را هماهنگ کند و تعارض را شفاف ثبت کند.

# قوانین تغییر مستندات

- هر تغییری در سیستم‌ها باید در فایل md مربوطه ثبت شود.
- هر فایل مستندات باید در انتهای خود بخش Changelog داشته باشد.
- تغییرات مهم باید همراه تاریخ ثبت شوند.
- تغییرات معماری باید در ../architecture/ARCHITECTURE.md نیز منعکس شوند.
- تغییرات عمومی پروژه باید در فایل Changelog اصلی پروژه نیز ثبت شوند، اگر چنین فایلی وجود داشته باشد.
- Codex حق ایجاد تغییر در سیستم‌ها، معماری یا قوانین پروژه بدون بروزرسانی مستندات مرتبط را ندارد.
- مستندات باید عملی، قابل استفاده برای توسعه، و هماهنگ با وضعیت واقعی پروژه باشند.

# قوانین کدنویسی

- کد باید بر اساس اصول Clean Code نوشته شود.
- معماری پروژه باید Modular باشد و از وابستگی‌های غیرضروری بین بخش‌ها جلوگیری کند.
- استفاده از Design Pattern فقط زمانی مجاز است که به سادگی، توسعه‌پذیری یا نگهداری بهتر سیستم کمک کند.
- Business Logic باید از Controller، View و UI جدا نگه داشته شود.
- در صورت نیاز باید از Service، Action، DTO، Policy، Event، Job، Queue، Interface و لایه‌های مشابه استفاده شود.
- ساختار کد باید قابل توسعه باشد، چون امکانات پروژه ممکن است در آینده گسترش پیدا کنند، کاهش یابند یا تغییر کنند.
- قوانین تجاری نباید به صورت hard-coded و پراکنده در کد نوشته شوند.
- منطق تکراری باید تا حد امکان حذف یا در ساختارهای قابل استفاده مجدد متمرکز شود.
- عملیات مهم باید در صورت نیاز log شوند.
- تمام منطق‌های مالی باید قابل ردیابی، قابل بررسی و قابل audit باشند.

# قوانین معماری

- پروژه باید Dockerized باشد.
- معماری باید برای رشد آینده و افزایش حجم کاربران، محصولات، سفارش‌ها و سرویس‌ها آماده باشد.
- پروژه باید از ابتدا برای Admin Panel، User Panel، Product Management، Order Management، Payment Control، Profit Calculation، Accounting Integration و Product Import آماده طراحی شود.
- سیستم باید به عنوان یک پلتفرم واقعی کسب‌وکار طراحی شود، نه فقط یک فروشگاه ساده.
- هدف عملی اولیه، راه‌اندازی ساختار پایه پروژه شامل Authentication، Admin Panel، User Management و زیرساخت فنی تمیز است.
- معماری باید امکان اضافه شدن سرویس‌های آینده مانند Node.js services، queue workers، automation services و integration services را داشته باشد.

# قوانین منطق مالی

- سود وب‌سایت فقط باید از فروش‌هایی محاسبه شود که از طریق payment gateway وب‌سایت تکمیل شده‌اند.
- فروش‌های فروشگاه فیزیکی خارج از چرخه تقسیم سود وب‌سایت هستند.
- کالاهایی که توسط Peyman خریداری می‌شوند به صورت خودکار وارد محاسبه سود وب‌سایت نمی‌شوند.
- فقط محصولاتی که وارد چرخه فروش وب‌سایت شده‌اند باید در محاسبه سود لحاظ شوند.
- هزینه پایه محصول باید هنگام ورود محصول به چرخه فروش وب‌سایت ثبت و حفظ شود.
- افزایش قیمت بازار در آینده نباید به صورت مصنوعی باعث افزایش سود مشترک شود.
- سود مشترک باید بر اساس فروش واقعی وب‌سایت، هزینه پایه محصول و هزینه‌های تاییدشده مرتبط با وب‌سایت محاسبه شود.
- هزینه‌های بالاتر از 15,000,000 تومان، به جز هزینه خرید محصول، فقط در صورت تایید هر دو طرف وارد محاسبه سود می‌شوند.
- Nima مسئول زیان مالی یا دوره‌های سود منفی نیست.

# قوانین مدیریت تغییر

- قبل از تغییر هر سیستم مهم، Codex باید فایل md مرتبط با آن سیستم را بررسی کند.
- بعد از تغییر هر سیستم مهم، Codex باید فایل md مرتبط را بروزرسانی کند.
- هر فایل Markdown مستندات باید بخش Changelog داشته باشد.
- تصمیم‌های مهم معماری، مالی، عملیاتی و فنی باید در مستندات ثبت شوند.
- مستندات باید با تغییرات واقعی پروژه همگام بمانند و نباید از وضعیت پیاده‌سازی عقب بمانند.
